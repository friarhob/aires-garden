## Status

Design decisions recorded below.

## Context

The data side of multilingual content is solved: [ADR-0004](../../decisions/0004-multilingual-post-frontmatter-conventions.md) locks down localised slugs, the `Translation_key` group anchor, the directory layout, and the `Lang` format; `add-content-model` enforces the schema and the directory invariant via `frontmatter_lint`. What's missing is the rendering side ‚Äî Pelican templates and URL routing don't yet reflect that posts can exist in multiple languages.

Specifically: Pelican's native translation-linking matches articles by shared `Slug:`, but ADR-0004 rule 1 deliberately gives each language its own slug so URLs read native ("/ola-jardim/" in Portuguese, "/hello-garden/" in English). That decision came with a known cost ADR-0004 documented up front: "a small plugin is required to group articles by `Translation_key:` and populate `article.translations` for Jinja templates. That plugin belongs in the future `add-i18n-rendering` change." This change builds that plugin and the templates that consume it, alongside two adjacent concerns the brief lists for the same proposal: per-language RSS feeds and the language switcher in the header.

The existing PT post `ola-jardim` lives at the transitional URL `/ola-jardim-pt.html` (Pelican's default for non-default-language articles) ‚Äî `add-deploy-pipeline`'s task 6.5 explicitly flagged that the terminal `/ola-jardim/` form requires this change. The site is pre-launch and noindexed, so the URL change has no public consequence beyond updating the post's deployed URL once.

A separate concern surfaces at the same time: the multilingual 404. GitHub Pages serves exactly one `output/404.html` for every unknown URL with no per-language routing layer. Static HTML+CSS alone cannot read `window.location` and prioritise per-language content. [ADR-0007](../../decisions/0007-small-clientside-js.md) records the stance that a small inline JS is acceptable here because it meets all three of its bars (static-host necessity, graceful degradation when JS is off, inline and small). This change is the first concrete consumer of that ADR.

## Goals / Non-Goals

**Goals:**

- Templates render the multilingual data the schema already enforces: language switcher, "Available in" badges, per-language indexes, per-language RSS feeds.
- All language versions of a post live at the terminal `/<slug>/` URL form ADR-0004 specified ‚Äî no language suffix in the path, regardless of which lang is the default.
- Single-language posts appear on the homepage. The existing PT-only `ola-jardim` should be reachable from `/` without tricks.
- The 404 page shows the reader a localised message based on the URL they tried, with a static stacked fallback when JS is disabled (per ADR-0007 bar 2).
- Plugin code stays small, scoped, and split by concern: article translation grouping and 404 stitching are unrelated lifecycles and ship as separate plugins.

**Non-Goals:**

- **Other HTTP error pages (403, 5xx).** GitHub Pages only serves the custom `404.html` from our build artefact; every other error response is generated by GitHub's CDN/infrastructure with content we cannot customise. This is a host limitation, not a deferred feature. Migrating off GH Pages to a host with edge routing is rejected for this change (see [ADR-0005](../../decisions/0005-dns-stays-on-gandi.md)).
- **Tag pages.** Owned by the next-up `add-tags-and-drafts` change. The per-language index template introduced here will be reused there, but the tag pages themselves are not built in this change.
- **Gallery / visual design overhaul of the homepage.** The dedupe-by-`Translation_key` aggregate is implemented as a minimal extension of the existing `index.html`, not a redesign. The full gallery (cards with cover images, typography, etc.) is owned by `add-design-tokens`.
- **Persistence of reader's preferred language** (e.g. via `localStorage`, cookies). The brief and the URL structure (`/<lang>/` indexes, in-page language switcher) cover the use case without persistence; per ADR-0007 example list, this would not pass the static-host-necessity bar.
- **Auto-translation tooling.** Translations are human-authored. The plugin reads what's there; it doesn't generate anything.
- **Search.** No search index is built or shipped.
- **`launch-to-apex` URL redirects.** The URL break for `ola-jardim` (`/ola-jardim-pt.html` ‚Üí `/ola-jardim/`) is a one-time deliberate change while pre-launch and noindexed; no `301` infrastructure is set up. Post-launch concerns are the `launch-to-apex` change's job.

## Decisions

### D1: Two plugins, not one

`plugins/i18n_grouping/` and `plugins/multilingual_404/` ship as separate Pelican plugins. The article-grouping plugin runs every build via `article_generator_finalized`, walks `generator.articles + generator.translations`, builds a `dict[translation_key, list[Article]]`, and attaches each article's group to it as `article.translation_group` (a list of sibling articles, including itself). The 404 plugin runs once via `page_generator_finalized` (or `finalized`, depending on Pelican's signal ordering ‚Äî verified during implementation), reads `content/pages/404.<lang>.md` files, renders each body, embeds them in a single `output/404.html` with the slug-to-lang map and inline JS.

Two plugins, not one combined "i18n" package, because:

- Their lifecycles don't overlap. Grouping reads articles; 404 reads pages and writes one HTML file. Zero shared state.
- Their consumers differ. Article templates consume `article.translation_group`; the 404 plugin's output is consumed by readers via the static file. A future change wanting article grouping (e.g. tag-page grouping in `add-tags-and-drafts`) imports just `i18n_grouping`; bundling the 404 stitcher in would force that change to drag in unrelated code.
- Failure isolation. A bug in the 404 builder cannot corrupt article grouping if they're separate modules.

The cost is one extra entry in `PLUGINS` and a few extra lines of boilerplate. Cheap.

**Alternatives rejected:**

- **One combined plugin (`plugins/i18n/`).** Reads cleaner under one "i18n" name in `PLUGINS`, but conflates two unrelated lifecycles and complicates future code that wants only one of them.
- **Combined plugin with internal modules.** Worst of both worlds: package name still imprecise, internal indirection without the clean separation a real package split provides.

### D2: Lang list is computed dynamically; no `KNOWN_LANGS` setting

The language switcher and the per-language index generator both need "what langs does this site have." Three options:

1. Hardcode a list in templates (`{% for lang in ['en','pt','es','fr'] %}`).
2. New Pelican setting `KNOWN_LANGS = ['en', 'pt', 'es', 'fr']`.
3. Compute dynamically as the union of `Lang` values across all articles.

We pick (3). `frontmatter_lint` already validates each `Lang` against `langcodes.tag_is_valid` at write time, so by the time the build runs, every value in content is a real ISO 639-1 code by construction. Adding a French post auto-extends the switcher and creates `/fr/` with no config edit. The dynamic computation is one set comprehension on `generator.articles + generator.translations`.

**Alternatives rejected:**

- **Hardcoded template list:** drift surface ‚Äî adding a language requires editing both content and templates.
- **`KNOWN_LANGS` setting:** explicit but redundant given the lint already gates `Lang`. The setting would either need to match the union of langs in content (and drift if the author forgets) or be a strict superset (and add disabled buttons for langs that have no posts ‚Äî visual noise).

### D3: `ARTICLE_LANG_URL = "{slug}/"` for terminal-form URLs

Pelican's defaults split URL templates by lang: `ARTICLE_URL` for `DEFAULT_LANG` articles, `ARTICLE_LANG_URL` for non-default. We set both to the same template:

```python
ARTICLE_URL       = "{slug}/"
ARTICLE_SAVE_AS   = "{slug}/index.html"
ARTICLE_LANG_URL  = "{slug}/"
ARTICLE_LANG_SAVE_AS = "{slug}/index.html"
```

Because each language already has a unique localised slug per ADR-0004 rule 1, the URLs don't collide ‚Äî `/ola-jardim/` and `/hello-garden/` are different paths even though both use the same template. This is the terminal form ADR-0004 specifies and the deploy-pipeline's task 6.5 forward-pointed to.

**Alternative rejected:** Keep Pelican's default `ARTICLE_LANG_URL = "{slug}-{lang}.html"`. Reads as "/ola-jardim-pt.html" ‚Äî privileges the default language in URL aesthetics, contradicts ADR-0004, and surfaces lang as a URL-visible second-class signal.

### D4: Homepage deduplicates by `Translation_key`

The default Pelican index iterates `generator.articles` (default-lang articles only). Under our model, that hides single-language non-default-lang posts entirely (`ola-jardim` would not appear on `/`). The fix:

- The `i18n_grouping` plugin walks `generator.articles + generator.translations`, groups by `Translation_key`.
- The aggregate index template iterates the *groups*, one card per `Translation_key`. The card links to the group's "preferred" article: the one in `DEFAULT_LANG` if available, else the first article in the group (sorted by `Lang` alphabetically ‚Äî deterministic, no implicit "first written" tracking).
- Each card displays the languages the group has (links to each translation).

**Alternatives rejected:**

- **Iterate `generator.articles` only.** Simplest, but hides single-language non-default-lang posts.
- **One card per article (each translation gets a card).** Inflates the homepage with duplicate cards of the same logical post.

### D5: Per-language indexes at `/<lang>/`

For each lang in the dynamic union (D2), generate a `<lang>/index.html` listing articles with that `Lang` value, sorted by `Date` descending. Implemented either via Pelican's `DIRECT_TEMPLATES` mechanism (if it cleanly accepts a per-lang variant) or via a small generator hook in `i18n_grouping` that emits one HTML file per lang. The per-lang template (`themes/garden/templates/lang_index.html`) is a near-duplicate of the aggregate `index.html` but iterates a filtered list rather than groups.

**Alternative rejected:** No per-language indexes; only the aggregate `/`. The brief and the language switcher's "non-post page" target both rely on `/<lang>/` existing. Without it, the switcher would have nowhere to point on non-post pages.

### D6: Language switcher behaviour

Lives in `themes/garden/templates/base.html`. The switcher is conceptually a *scope picker* rather than a pure language picker: it always shows an "All" entry that points to the aggregate homepage `/`, followed by one entry per lang in the dynamic union.

**The "All" entry:**

- Always present, always enabled, always links to `/`.
- Highlighted (`class="lang-current"`) only when the current page is `/` itself.
- Label is the text string "All". A future icon/symbol substitution is a design-tokens concern (`add-design-tokens`); locking iconography here would create something to revisit then. Plain text is also screen-reader-friendly and free of emoji-rendering inconsistency.

**Per-language entries:**

- **On post pages** (when the template has access to `article`): if the article's `translation_group` contains a sibling with that `Lang`, render an enabled link to that sibling's URL; else render a visibly disabled element (e.g. `<span class="lang-disabled">`). The current article's lang is highlighted.
- **On non-post pages** (homepage `/`, per-lang indexes, 404, future tag/page templates): always render an enabled link to `/<lang>/`. There's no per-page concept of "missing translation" ‚Äî the per-lang index always exists for any lang the site has. The current page's lang (if any) is highlighted.

**State table:**

| Page type | "All" entry | Per-lang entries |
|---|---|---|
| Post page | enabled ‚Üí `/` | each links to translation; missing disabled; article's lang highlighted |
| Aggregate `/` | enabled ‚Üí `/`, highlighted | each links to `/<lang>/` |
| Per-lang `/<lang>/` | enabled ‚Üí `/` | each links to `/<lang>/`; current lang highlighted |
| 404 | enabled ‚Üí `/` | each links to `/<lang>/` |

**Alternatives rejected:**

- **Hide the switcher entirely on non-post pages.** Simpler logic but loses discoverability of "this site has these languages" for readers who land on the homepage. The brief explicitly calls for the switcher in the header.
- **No "All" entry; rely on the site-name link in the header to reach `/`.** The site-name link already exists, so structurally `/` is reachable. But putting "All" inside the switcher makes the scope semantics explicit ‚Äî readers see "All / EN / PT" as parallel choices rather than treating "All" as a hidden side-effect of clicking the site name.
- **"All" entry as an icon (üåê, ‚â°, ‚Ä¢).** Cleaner visually, worse for screen readers (emojis announce as "globe emoji"), and locks in iconography that `add-design-tokens` is the better venue to decide.

### D7: Multilingual 404 ‚Äî per-lang source files merged at build time

Authors create `content/pages/404.<lang>.md` (one per language); the plugin merges. Per [ADR-0007](../../decisions/0007-small-clientside-js.md), the merger emits a single `output/404.html` containing all translations as `<section data-lang="<lang>">` blocks plus inline JS that prioritises one section based on URL inference.

**Authoring shape:**
```
content/pages/
‚îú‚îÄ‚îÄ 404.en.md
‚îî‚îÄ‚îÄ 404.pt.md
```

Each file's frontmatter is the page schema (Title, Status: hidden, Lang). The merger discards the per-file `Save_as` / `URL` (those live on the merged output, not on the parts).

**Output shape (skeleton):**

```html
<!DOCTYPE html>
<html lang="en" data-default-lang="en">
<head>...</head>
<body data-i18n="404">
  <section data-lang="en">‚Ä¶rendered EN body‚Ä¶</section>
  <section data-lang="pt">‚Ä¶rendered PT body‚Ä¶</section>
  <script>/* inline JS, see D8 */</script>
</body>
</html>
```

**Alternative rejected:** Single `404.md` with all langs stacked manually inside one file, with marker tags (`<section data-lang="‚Ä¶">`) in the Markdown body. Requires no plugin code, but conflates per-language content authoring into one file ‚Äî fragile when adding a new lang, easy to miss.

### D8: 404 language inference is two-stage; build-time slug‚Üílang map

Inline JS (~15 lines) runs on `DOMContentLoaded`. Stage 1: regex-match the URL path against `^/([a-z]{2})/`. If the captured lang is in the document's `<section data-lang>` set, prioritise that section. Stage 2 (only if stage 1 didn't match): match the URL path against `^/([^/]+)/?$` and look up the captured slug in a build-time `SLUG_LANGS` JSON object embedded in a `<script type="application/json">` block inside the 404. If the slug exists in the map, use its lang; else fall back to the document's `data-default-lang` attribute.

"Prioritise" means: hide all `<section data-lang>` blocks except the matched one, and set `<html lang>` to that lang. With JS disabled, all sections remain visible (graceful degradation per ADR-0007 bar 2). The script is inline (no external file, no build artefact). The slug-to-lang map is generated at build time by the `multilingual_404` plugin from `generator.articles + generator.translations`.

**Alternatives rejected:**

- **Stage 1 only (URL prefix only).** Loses the `/<localised-slug>-typo/` case where the typo is close to a known PT slug but doesn't carry `/pt/` prefix. Slug map adds little code and improves the matching for free.
- **Edit-distance fuzzy matching.** Returns "did you mean‚Ä¶?" suggestions but blows up complexity. Out of scope; future could add it as a third stage if the value is real.
- **Server-side language detection (`Accept-Language` header).** Not available on GH Pages; the static host can't read request headers.

### D9: RSS ‚Äî per-language feeds + aggregate

Set `TRANSLATION_FEED_ATOM = "feeds/{lang}.atom.xml"` in `pelicanconf.py`. Pelican generates one Atom feed per non-default `Lang`. The default language's feed is generated by `FEED_ALL_ATOM = "feeds/all.atom.xml"` (already configured in `publishconf.py`) ‚Äî but that's the *aggregate*, not a default-lang feed. To get a true per-default-lang feed (`feeds/en.atom.xml`), the cleanest path is to also set `FEED_ATOM = "feeds/{lang}.atom.xml"` so Pelican emits one for the default lang too, alongside the aggregate.

Verified Pelican behaviour during implementation. If `FEED_ATOM` and `TRANSLATION_FEED_ATOM` together don't produce a clean per-lang set, fall back to a small generator hook in `i18n_grouping` that emits per-lang feeds from the grouped articles.

**Alternative rejected:** Per-language feeds only, drop the aggregate `all.atom.xml`. Saves one feed file but breaks the existing publishconf-defined aggregate. Subscribers who want everything would have to subscribe to N per-lang feeds.

### D10: Card link strategy ‚Äî default-lang preferred, else alphabetical first

When the homepage card represents a `Translation_key` group, it links to: `default_lang_article` if present in the group; else the article whose `Lang` is alphabetically first (e.g. `en` < `pt`). Determinism matters for reproducible builds; alphabetical is the simplest deterministic rule and avoids tracking "which language was written first" (which `Translation_key` already encodes structurally but not semantically per article).

**Alternative rejected:** Link to whichever article was written first (most recent? oldest? ‚Äî `Translation_key`-as-anchor language?). Adds a "first language" implicit ordering that isn't recorded in metadata; alphabetical is just as good and trivially derivable.

## Risks / Trade-offs

- **Pelican signal ordering for the 404 plugin** ‚Üí the page generator finalises pages, but the merged 404 needs to write *after* per-page rendering finishes. **Mitigation:** use `page_generator_finalized` or `finalized`, verified during implementation. If neither propagates cleanly, fall back to a custom generator class registered via `get_generators`.

- **`TRANSLATION_FEED_ATOM` quirks** ‚Üí Pelican's translation feed support has historical inconsistencies (some versions emit feeds only for non-default-lang, some respect `FEED_ATOM` interaction differently). **Mitigation:** verify against Pelican 4.10 during implementation; if the standard settings don't produce one feed per lang cleanly, emit feeds via a `i18n_grouping` plugin hook from the grouped articles.

- **URL break for `ola-jardim`** ‚Üí `/ola-jardim-pt.html` ‚Üí `/ola-jardim/`. Anyone who linked the old URL gets a 404. **Mitigation:** the site is pre-launch and noindexed (deploy-pipeline noindex stays active), so external link breakage is bounded to "people who manually shared the URL." Recorded for the `launch-to-apex` checklist.

- **`SLUG_LANGS` map bloats the 404 page on a large garden** ‚Üí For ~50 posts, the map is ~1 KB JSON. For 1000+ posts, it's still tiny. No mitigation needed at expected scale; if the garden ever grows large enough that the 404 ships hundreds of KB, revisit.

- **JS-disabled readers see all language sections stacked on the 404** ‚Üí By design (ADR-0007 bar 2). The fallback is functional, not pretty.

- **`<html lang>` on the 404 is set client-side by JS** ‚Üí Without JS, `<html lang="en">` (the default-lang attribute) is what screen readers see, even if the user typed a `/pt/...` URL. Acceptable degradation.

- **Two plugins instead of one means two `register()` functions and two `PLUGINS` entries** ‚Üí minor boilerplate cost, accepted in D1.

- **Pelican 4.x's translation-grouping with localised slugs is non-default territory** ‚Üí `i18n_grouping` is the load-bearing piece. If a future Pelican upgrade changes the relevant signals or article model, this plugin is the migration touchpoint. **Mitigation:** the plugin is small (~50 LoC) and the seam is well-defined (`article.translation_group` attribute).

- **The slug-to-lang map leaks information about the site structure to anyone who reads the 404** ‚Üí Negligible; the same slugs are public via the homepage and feeds anyway. Not a security risk.

## Migration Plan

In-change migration only. Order:

1. Land both plugins (`i18n_grouping`, `multilingual_404`) and register in `pelicanconf.py`. Verify build still succeeds.
2. Update Pelican settings (`ARTICLE_LANG_URL`, `ARTICLE_LANG_SAVE_AS`, `TRANSLATION_FEED_ATOM`, `FEED_ATOM`).
3. Update templates (`base.html`, `index.html`, `article.html`, new `lang_index.html`, new `404.html` template used by the 404 merger).
4. Migrate `content/pages/404.md` ‚Üí `404.en.md` (rename, no body change). Add `content/pages/404.pt.md` (new file with PT body).
5. Run `make build` locally: verify the homepage shows `ola-jardim` once with PT badge, the post is at `/ola-jardim/`, the per-lang `/en/` and `/pt/` indexes exist, the feeds at `/feeds/all.atom.xml`, `/feeds/en.atom.xml`, `/feeds/pt.atom.xml` are present, and the merged `/404.html` contains both EN and PT sections.
6. Visit `/404.html` directly and verify the JS prioritises one section based on inferred lang (test via browser dev console with synthetic URLs).
7. Run `openspec validate add-i18n-rendering --strict`.

Rollback: revert the change. Plugin code lives in its own files; templates are isolated to `themes/garden/templates/`; settings can be removed without affecting content. The renamed `404.en.md` would need to be renamed back to `404.md` to fall back to the pre-change behaviour.
